# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

`shinyGolem` is an R package built with the {golem} framework that wraps a Shiny application (an Iris dataset visualization demo). Because it's a golem app, the app itself *is* an R package — it's developed, tested, and built the same way as any other R package (`devtools`, `testthat`, `R CMD check`), and then run via `run_app()`.

## Common commands

All commands are run from R (RStudio console or `Rscript`), not a shell build tool.

- **Run the app locally (dev mode):** `source("dev/run_dev.R")` — sets `golem.app.prod = FALSE`, detaches/reloads the package via `golem::document_and_reload()`, then calls `run_app()`. Rerun this after every source change to reload and relaunch.
- **Load/reload package without running app:** `golem::document_and_reload()` or `devtools::load_all()`.
- **Run the full test suite:** `devtools::test()`.
- **Run a single test file:** `devtools::test_active_file("tests/testthat/test-mod_data_viz_demo.R")` or `testthat::test_file("tests/testthat/test-mod_data_viz_demo.R")`.
- **Run a single test by name:** `testthat::test_file(..., filter = "app ui")` or use `devtools::test(filter = "shinytest2")`.
- **Full package check (mirrors CI):** `devtools::check()`.
- **Build the package tarball:** `devtools::build()`.
- **Update DESCRIPTION dependencies from code:** `attachment::att_amend_desc()`.
- **Spelling check:** `Rscript tests/spelling.R` (uses `inst/WORDLIST` for allowed words).
- **Deploy (manual):** `rsconnect::deployApp()` (see `dev/03_deploy.R`; CI does this automatically via `dev/deploy_cicd.R`).

There is no `package.json`/Makefile — everything goes through R. `testGolem.Rproj` is the RStudio project file.

## Architecture

### golem app structure
- `R/run_app.R` — `run_app()` is the sole exported function; wraps `shiny::shinyApp(ui = app_ui, server = app_server)` in `golem::with_golem_options()`.
- `R/app_ui.R` — top-level UI (`app_ui()`), a `navbarPage` that assembles modules into tabs, plus `golem_add_external_resources()` for JS/CSS/favicon bundling from `inst/app/www`.
- `R/app_server.R` — top-level server (`app_server()`), which calls each module's `*_server()` function.
- `R/mod_*.R` — Shiny modules, each with a paired `mod_<name>_ui(id)` and `mod_<name>_server(id)` function, following the standard golem module pattern (`NS(id)` namespacing). Currently: `mod_data_viz_demo.R` (Iris data viz — filtering, scatter/smooth plot, boxplot, table).
- `R/app_config.R` — `app_sys()` (locate files inside the installed package via `system.file(..., package = "shinyGolem")`) and `get_golem_config()` (reads `inst/golem-config.yml` via the `config` package, driven by `GOLEM_CONFIG_ACTIVE`/`R_CONFIG_ACTIVE` env vars, default `"default"`).
- `R/golem_utils_ui.R` / `R/golem_utils_server.R` — golem's standard UI/server helper functions (tag manipulation, `%||%`, `col_*` column wrappers, etc.), generated by `golem::use_utils_ui()`/`use_utils_server()`.
- `inst/golem-config.yml` — per-environment config (`default`, `production`, `dev`); notably `app_prod` (prod flag) and `spinner_type_app` (busy-indicator spinner style, read in `mod_data_viz_demo_ui`).
- `inst/app/www` — static web assets, added via `add_resource_path("www", ...)`.
- `app.R` — production launch entrypoint used by rsconnect/Docker (`pkgload::load_all()` + `options(golem.app.prod = TRUE)` + `run_app()`). This file is excluded from the built package (`.Rbuildignore`) but required for deployment.
- `dev/` — golem's numbered workflow scripts (`01_start.R`, `02_dev.R`, `03_deploy.R`) plus `run_dev.R` (local dev launcher) and `deploy_cicd.R` (CI deploy script, reads `RSCONNECT_USER`/`RSCONNECT_TOKEN`/`RSCONNECT_SECRET` from env). These are templates/checklists golem scaffolds — treat them as a log of setup steps, not code to run wholesale.

### New modules
Use `golem::add_module(name = "...", with_test = TRUE)` to scaffold a new module (creates `R/mod_<name>.R` and a matching test file). Wire it up by adding `mod_<name>_ui(...)` to `app_ui.R` and calling `mod_<name>_server(...)` in `app_server.R`.

### Config values
Read config via `get_golem_config("some_value")`, not `config::get()` directly — this indirection handles `app_sys()` path resolution and the active-environment fallback logic.

## Testing

- Test framework: `testthat` (edition 3, per `Config/testthat/edition` in DESCRIPTION).
- `tests/testthat/test-golem-recommended.R` — golem's scaffolded baseline tests: checks `app_ui`/`app_server` formals aren't broken, config values resolve correctly per environment, and a `golem::expect_running()` smoke test that the app actually launches.
- `tests/testthat/test-mod_*.R` — per-module tests, typically using `shiny::testServer()` for server logic and `golem::expect_shinytaglist()` for UI.
- `tests/testthat/test-shinytest2.R` / `test-app-function.R` — end-to-end browser tests via `{shinytest2}`'s `AppDriver`, including snapshot assertions (`expect_values()`, `expect_screenshot()`). Snapshots live under `tests/testthat/_snaps`. `tests/testthat/setup-shinytest2.R` loads app support files for these. Regenerate/update screenshots with the normal `testthat` snapshot workflow (`testthat::snapshot_review()` / `snapshot_accept()`) — do not hand-edit files in `_snaps`.
- Module server input IDs in tests are namespaced, e.g. `` `data_viz_demo_1-dataset` `` for the `dataset` input inside the `data_viz_demo_1` module instance.
- Skip CRAN-inappropriate tests (e.g. shinytest2 browser tests) with `skip_on_cran()`.

## CI/CD

Three GitHub Actions workflows (all triggered on push/PR to `main`/`master`):
- `R-CMD-check.yaml` — full package check across a matrix of OS/R-version combos (uses `devtools`-equivalent `r-lib/actions`).
- `test-coverage.yaml` — runs `covr::package_coverage()`, uploads to Codecov.
- `shiny-deploy.yml` — checks the package, then (only on push to `main`/`master`) deploys to shinyapps.io by running `Rscript dev/deploy_cicd.R`, using `RSCONNECT_USER`/`RSCONNECT_TOKEN`/`RSCONNECT_SECRET` secrets.

Docker: `Dockerfile` builds from `rocker/shiny:latest`, installs the package from GitHub via `remotes::install_github`, and runs `shinyGolem::run_app()` on port 3838 (configured through `Rprofile.site`).
